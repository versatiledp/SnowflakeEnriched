JSON data processing¶

JSON stands for JavaScript Object Notation. It is a minimal readable format for semi-structured data. It is a standardized format commonly used to transfer data as text that can be sent over a network. JSON is used by lots of APIs and databases to transmit data between servers and web applications. It is alternative to XML and easy for both humans and machines to read.

Snowflake allows loading JSON data directly into relational tables. We can query this data with SQL and join it to other data without doing any transformations. At load time, Snowflake parses the JSON based on the key-value pairs. Structural information is derived based on the schema definition embedded in JSON string at reading time (Schema-on-Read).

Enrollment files are in JSON format. These files contain information around enrollment rates in various institutions for the academic years 2017, 2018, and 2019. The enrollments are categorized by race, ethnicity, sex, etc.. We have loaded these files in the staging environment of the Snowflake.

Note

Please refer to the section 'Load the files to the internal stage for IPEDS ' in the chapter 'Introducing data platform ' for creating staged files.

The staged enrollment files are invoked using @IPEDS_EFFY. JSON data from the staged file is retrieved using the following code.

SELECT N.$1 JASONdata FROM  @IPEDS_EFFY N;

Each row holds one JSON document, ingested from the source into a staged file.

If there is a JSON object stored in a physical table called IPED Column jdata, we could read InstitutionIdentifier as below.

SELECT jdata:UNITID AS InstitutionIdentifier
   FROM IPED;

jdata = Column name in IPED table type VARIANT holding JSON object
UNITID = attribute in the JSON schema

Let us retrieve column InstitutionIdentifier from JSON object in the staged file.

SELECT S.$1:UNITID AS InstitutionIdentifier
    FROM @IPEDS_EFFY S;

$1 = positional attribute in staged file
An alias InstitutionIdentifier to column name

The column that was returned in the previous step comes back with double quotes around it. Now we are going to cast that data into an actual data type as shown below.

SELECT S.$1:UNITID::INTEGER
     AS InstitutionIdentifier FROM @IPEDS_EFFY S;

We used :: to cast to a INTEGER data type.

Transformation is performed while loading Enrollment.

The stored procedure pr_Enrollment_Load is used for loading the servicing layer table. The stored procedure loads data for specific years based on the parameter passed to it. Each time when this stored procedure is called, it deletes records from Enrollment for that year and then re-inserts them to Enrollment from that year's od_Enrollment.

We are interested in determining the distribution of data that has been imputed vs reported. To do so, we read the imputation column to obtain the total number of imputed and reported variable counts, which we store in VariablesReportedCount and VariablesImputedCount. Then, we calculate their percentages and store them in ImputationVarPercent and ReportedVarPercent, respectively.

The stored procedure script for pr_Enrollment_Load is available below. The data transformation needed to get ImputationVarPercent and ReportedVarPercent is performed in the same stored procedure.

  • The procedure is defined with argument.

  • Runtime language is Javascript.

  • The Procedure returns a string.

  • Multiple JavaScript variables are used.

  • Single SQL command/text is assigned to one JavaScript variable.

  • These variables are combined into a string array.

  • For loop is used to execute all SQL commands by calling snowflake.execute and passing the Javascript variable.

  • Check the results on each call of snowflake.execute.

Executing load procedures¶

Data is available in OD table and it is ready for further processing. Transformation logic is coded in the stored procedure (SP) to process specific academic year. Stored procedure deletes data from the servicing layer, if there exsists any data for the given academic year. Then we load data from the OD source.

We execute this procedure to load data for years 2017, 2018, and 2019.

TRUNCATE TABLE Enrollment;
CALL pr_Enrollment_Load(2017::FLOAT);
CALL pr_Enrollment_Load(2018::FLOAT);
CALL pr_Enrollment_Load(2019::FLOAT);

Summary¶

In this chapter, we analyzed the annual enrollment data of various institutions. We discussed the JSON-formatted data ingestion process. JSON data can be loaded directly into table columns of type VARIANT. These VARIANT columns are queried using SQL SELECT statements that reference JSON document elements by the hierarchical paths. We also explored building transformation logic in stored procedures to map additional attributes.

In most of the current big data platform, high volume data is ingested in the Hadoop Distributed File System (HDFS). In the next chapter, we will cover how to ingest ORC files coming from the HDFS system.
