ORC ingestion¶

Optimized Row Columnar(ORC) is a self-describing type-aware columnar file format designed for Hadoop workloads. It is optimized for large streaming reads. By storing data in a columnar format it allows user to read, decompress, and process only the values that are required for the current process. Snowflake manages AVRO and ORC files similar to semi-structured JSON data from VARIANT type.

Note

Please refer to the section ‘Load the files to the internal stage for IPEDS’ in the chapter ‘Introducing data platform’ for creating staged files.

We can convert VARIANT type column having ORC data to a multi-column table by specifying column names and data types. ORC data from the staged file @IPEDS_IC is retrieved using the following code.

SELECT O.$1 JASONdata FROM  @IPEDS_IC O;

ORC data is stored as a JSON document. JSON document processing can be used to transform ORC data.

We are using few Snowflake native functions to work with semi-structured JSON data.

Functions to manage semi-structured data¶

JSON data sets are a simple concatenation of multiple key-value pairs. In this section, we are going to cover few new Snowflake functions like FLATTEN(), ARRAY_SIZE(), LATERAL FLATTEN(), and PARSE_JSON().

These functions are used to query the semi-structured data and convert semi-structured data to a relational format.

FLATTEN() is a table function that extracts the objects and keys into a separate table. It takes a VARIANT, OBJECT, or ARRAY column and produces a lateral view.

PARSE_JSON() function takes a string as input and returns a JSON-compatible VARIANT.

ARRAY_SIZE() function returns size of the array in the VARIANT value.

LATERAL FLATTEN() is used to flatten the data and to extract array from JSON document.

The function ARRAY_SIZE() returns the size of an array in the VARIANT value. In this case, it contains 2 arrays: GoalsStatistics and Division.

ARRAY_SIZE(PLAYER_DATA:GoalsStatistics)

LATERAL FLATTEN() is used to extract an array from JSON data. The query below uses LATERAL FLATTEN() to take the items in the list from PLAYER_DATA: GoalsStatistics and make them their own dataset. INPUT, VALUE, INDEX are reserved keywords. It is important to give alias GoalsStat to flatten results.

SELECT
PLAYER_DATA:Name::STRING "Player Name",
GoalsStat.VALUE AS Goals,
GoalsStat.INDEX
FROM S_PLAYERS,
    LATERAL FLATTEN(INPUT =>
        PLAYER_DATA:GoalsStatistics) GoalsStat;

We can have multiple FLATTEN functions within the same query as shown below. Here the query returns a list of arrays from GoalsStatistics and Division data objects.

SELECT
PLAYER_DATA:Name::STRING "Player Name",
GoalsStat.VALUE AS Goals,
Division.VALUE AS Divisions
FROM S_PLAYERS,
    LATERAL FLATTEN(INPUT =>
        PLAYER_DATA:GoalsStatistics) GoalsStat,
    LATERAL FLATTEN(INPUT =>
        PLAYER_DATA:Division) Division;

We are joining Division with Season using LATERAL FLATTEN and a left OUTER join. The second FLATTEN clause is used to flatten the Season array within the flattened Division array and retrieve the Season played by each Division. PARSE_JSON is used to create a JSON VARIANT out of the Division and Season arrays joined using a LEFT OUTER join.

SELECT
    d.PLAYER_DATA:Name::STRING "Player Name",
    c.value:Name::STRING "Division Name",
    s.value::STRING Season
FROM
    S_PLAYERS d,
    LATERAL FLATTEN(INPUT =>
     PARSE_JSON(d.PLAYER_DATA:Division),OUTER => true) c,
    LATERAL FLATTEN(INPUT =>
     PARSE_JSON(c.value:Season),OUTER => true) s

Institutional charges¶

This section covers data ingestion concerning tuition fees charged by institutions for various courses and branches during the 2017, 2018, and 2019 academic years. The data files received for institutional charges are in ORC format.

The dataset also categorizes fees for graduate programs like Medicine, Dentistry, Pharmacy, Law, etc., as a different set of attributes within academic institutions charge records.

Fees are also categorized for In-State and Out-of-State students. Expenses such as books/supplies, boarding, and resources, are also available in these data files.

The information in this large file is broadly categorized into 4 groups described below.

Institutional charge (General)

This cross-section of attributes covers average tuition/required fees, and per-credit fees charged by educational institutes. Fees vary significantly from state to state, public to private, and ranking of institutes (demand vs supply).

Institutional charge by academic branch

The details of the fees by academic subjects like dentistry, medicine, optometry, osteopathic medicine, pharmacy, podiatry, veterinary medicine, and law by institutions for in-state and out-of-state students are available in this section.

Institutional charge by residency type

The residency type determines different fee structures in public universities for the same courses. Normally, in-state students are charged less than out-of-state. This set of attributes will provide insight into those patterns.

Institutional charge by category

Various fees like on-campus boarding, books/supplies, and other on-campus expenses are available in this section of the file.

Unpivot data to load into the servicing layer¶

In our case, data is stored in the OD layer in a pivoted record structure. We will need to unpivot these attributes to load the servicing layer.

We can unpivot data structure in a traditional SQL environment by creating a UNION on underlying attributes.

The following views are created to transform columns to rows. These views will be used in stored procedures.

The view v_od_InstitutionalChargeByAcademicBranch is created to convert columns from the underlying OD dataset to rows for the data processing layer. We store JSON objects in the OD layer to simplify the load. As new attributes are added, it is easy to manage data ingestion. JSON data structure is easy to consume in the application layer, as we will further detail in the next section.

Change in institutional charge operational datastore

The staged file data is grouped into three JSON objects:

  • Charges by the academic branch

  • Charges by residency type

  • Charges by category

These JSON objects are added as three columns in the OD table of data type VARIANT.

Views for servicing layer

The JSON VARIANT data type is used to re-create three views:

  • v_od_InstitutionalChargeByAcademicBranch

  • v_od_InstitutionalChargeByResidencyType

  • v_od_InstitutionalChargeByCategory

Show/Hide script for: Create OD InstitutionalChargeByAcademicBranch view for an alternative approach

DROP VIEW IF EXISTS v_od_InstitutionalChargeByAcademicBranch;
CREATE OR REPLACE VIEW
        v_od_InstitutionalChargeByAcademicBranch AS

SELECT AcademicYear,
InstitutionIdentifier,
LEFT(AcademicBranch.KEY
        ,POSITION('_', AcademicBranch.KEY, 1) - 1) AS
                                BranchName,
RIGHT(AcademicBranch.KEY, LEN(AcademicBranch.KEY)
        - POSITION('_',AcademicBranch.KEY, 1)) AS ChargeType,
AcademicBranch.VALUE AS Amount
FROM Od_InstitutionalCharge
        ,LATERAL FLATTEN(INPUT = > InstitutionalChargeByAcademicBranch)
        AcademicBranch;

Show/Hide script for: Create OD InstitutionalChargeByResidencyType view for an alternative approach

DROP VIEW IF EXISTS v_od_InstitutionalChargeByResidencyType;

CREATE OR REPLACE VIEW
        v_od_InstitutionalChargeByResidencyType AS
SELECT  AcademicYear,
InstitutionIdentifier,
LEFT(ChargeByResidencyType.Key,
        POSITION('Year',ChargeByResidencyType.Key,1)-1)
                AS ResidencyType,
RIGHT(ChargeByResidencyType.Key,1) AS PublishedYear,
ChargeByResidencyType.VALUE AS Amount
FROM Od_InstitutionalCharge,
          LATERAL FLATTEN(INPUT => InstitutionalChargeByResidencyType)
                ChargeByResidencyType

InstitutionIdentifier,ChargeByCategory.Key,

                AS CategoryType,
RIGHT(ChargeByCategory.Key,1) AS PublishedYear,
ChargeByCategory.VALUE AS Amount
FROM Od_InstitutionalCharge,
         LATERAL FLATTEN(INPUT => InstitutionalChargeByCategory)
                ChargeByCategory

Data load into the servicing layer¶

We will be using the OD table od_InstitutionalCharge and the above three views to finally land data to its destination. For the given year, these procedures will delete existing data from the destination tables and transform and load data from the OD layer.

Summary¶

Ingesting ORC files in Snowflake is as easy as ingesting JSON data. We learned about a variety of transformations, moving data from the landing layer to the servicing layer. We also covered how to unpivot data stored in the OD layer and load into the servicing layer. Finally, we detailed an alternative approach using the JSON VARIANT data type to perform transformations.

Spark executes the code much faster by caching data in memory across multiple parallel operations, whereas HDFS and MapReduce involve more reading and writing from disk. Along with Spark, the industry also adopted writing these dataframes into Parquet format. In the next chapter, we will be creating a Snowflake pipeline to ingest Parquet format files.
