Parquet ingestion¶

Apache Parquet is a column-oriented binary file format that provides optimizations to speed up the process and is a far more efficient file format than CSV or JSON. It is known for its features like performance, data compression, and ability to handle a wide variety of encoding types. Parquet files enable to fetch only the required columns and values, load those in memory and answer the query. If a row-based file format like CSV was used, the entire table would have to have been loaded in memory, resulting in increased I/O and worse performance. Snowflake manages Parquet files similar to semi-structured JSON data from VARIANT type.

We can convert VARIANT column Parquet data to a multi-column table by specifying column names and data types. Parquet data is retrieved from @IPEDS_ADM staged file using the following code.

Note

Please refer to the section ‘Load the files to the internal stage for IPEDS’ in the chapter ‘Introducing data platform’ for creating staged files.

SELECT P.$1 Parquetdata FROM  @IPEDS_ADM P;

The picture below shows Parquet data in the internal staged file.

Parquet data is also stored as JSON documents. JSON document processing can be used to transform Parquet data.

  • Execute the stored procedures.

Admission statistics data load¶

The service layer/fact table AdmissionStat is loaded from the OD table od_AdmissionStat by year. Every time this fact table is updated, it is loaded with the latest data for that year. Therefore, old data loaded for that year is deleted and new data is loaded from the OD table. We created a Javascript-based stored procedure pr_AdmissionStat_Load, in which we use 2 variables, sql_DelCommand, and sql_Command. The variable sql_DelCommand contains the SQL commands to delete the records from the fact table for the year passed to the stored procedure. The other variable sql_Command contains the main logic for updating the fact table for that year.

We have used a LEFT OUTER JOIN with AcademicInstitution on InstitutionIdentifier to get the matching surrogate key from the dimension table AcademicInstitution. If we do not find the matching record, we assign -1 to the corresponding entry of AcademicInstitutionUniqueDWSID in the fact table. To achieve this, we use the IFNULL command as IFNULL(D.AcademicInstitutionUniqueDWSID, - 1).

Here is the Snowflake stored procedure for processing the AdmissionStat table.

Show/Hide script for: Create AdmissionStat load stored procedure

var sql_Command =

FirstTimeDegreeOrCertificateSeekingStudentsSubmittingACTScores,
PercentOfFirstTimeDegreeOrCertificateSeekingStudentsSubmittingACTScores,

SATMath75thPercentileScore,

ACTEnglish75thPercentileScore,
ACTMath25thPercentileScore,
ACTMath75thPercentileScore)

IFNULL(D.AcademicInstitutionUniqueDWSID, - 1),
S.InstitutionIdentifier,
S.AcademicYear ,
S.SecondarySchoolGPA,
S.SecondarySchoolRank,
S.SecondarySchoolRecord,
S.CompletionOfCollegePreparatoryProgram,

arrSqlCmd[1] = sql_Command

Summary¶

In this chapter, we went through the data loading process of Parquet files in Snowflake. This data processing architecture is similar to that of CSV files and is made easily consumable by Snowflake. We loaded 3 years of admission statistics files in the staged area. We added the OD table, and the fact table for the servicing layer.

Data engineers have to deal with multiple file layout situations along with CSV, JSON, ORC, and Parquet formats. Most notable examples is the file having multiple record layouts. In the next chapter, we will cover one of these situations, where a file has records for multiple destination entities.
